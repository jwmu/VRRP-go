# ADR-001: VRRP-go 安全加固与代码简化决策

## 状态
已接受 (Accepted)

## 日期
2026-04-06

## 上下文
在 VRRP-go 项目的早期开发阶段，代码库存在以下挑战：
1. **安全性不足**：缺乏输入验证（如 VRID 范围、接口名合法性）和完善的报文边界检查。
2. **容错性低**：网络初始化失败时使用 `panic`，导致库无法被上层应用优雅捕获错误。
3. **复杂度较高**：状态机事件处理逻辑存在冗余，部分报文解析代码使用了低效的手动拷贝。

## 决策
执行了一次全面的安全加固和代码简化重构：
1. **输入验证**：强制验证 VRID (1-255) 和网络接口名（仅限合法字符）。
2. **移除 Panic**：将所有网络连接构造函数改为返回 `(Result, error)` 模式，确保库的调用方可以管理生命周期。
3. **报文解析加固**：在 `FromBytes` 中增加严格的字节长度预校验，防止 slice 越界。
4. **逻辑扁平化**：引入 `shutdownAll` 辅助函数，统一状态转换时的资源清理逻辑。

## 备选方案

### 保持原始 Panic 逻辑
- **优点**：简单，能在开发阶段快速暴露致命错误。
- **缺点**：生产环境下会导致整个程序崩溃，无法用于容器化或高可用场景。
- **结论**：拒绝，必须向生产品质转换。

### 引入第三方 Web 框架进行验证
- **优点**：利用现成库（如 validator）。
- **缺点**：增加不必要的外部依赖，VRRP 作为底层网络工具应保持极简。
- **结论**：拒绝，采用原生 Go 实现轻量验证。

## 后果
- 代码健壮性提升，消除了多个潜在的 Segfault 和 DoS 攻击向量。
- 库的 API 发生了破坏性变更（构造函数现在返回 error），相关调用方需要同步更新。
- 状态机代码行数减少，可读性增强，后续添加新协议特性时难度降低。
